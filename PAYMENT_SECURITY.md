# Dikshu's Cafe Secure Payment Security Architecture Blueprint

This blueprint describes the cryptographic and architectural controls integrated into the Dikshu's Cafe order fulfillment and payment verification engine.

---

## 1. Prevention of "Fake Payment Success" Exploits
*   **The Hazard**: An attacker intercepting client-side execution, modifying the browser's JavaScript variables or server response payloads to simulate a successful transaction event (e.g. force-forcing `status: "paid"` on order completion).
*   **Our Solution**: The **Two-Legged Backend Verification Protocol**:
    1.  The client-side Razorpay context *never* updates any database attributes upon payment success. Instead, the frontend receives three unique payment credentials: `razorpay_order_id`, `razorpay_payment_id`, and `razorpay_signature` (the verified cryptographic seal generated by Razorpay's Secure HSMs).
    2.  The client dispatches these coefficients via `POST /api/payments/verify`.
    3.  The server fetches the secret `RAZORPAY_KEY_SECRET` exclusively from private environment variables (`process.env.RAZORPAY_KEY_SECRET`).
    4.  The server computes a strict, one-way SHA256 HMAC hash of the combination:
        ```text
        Message = razorpay_order_id + "|" + razorpay_payment_id
        ExpectedSignature = HMAC_SHA256(Message, RAZORPAY_KEY_SECRET)
        ```
    5.  Using a constant-time string comparison, the server verifies `ExpectedSignature === razorpay_signature`.
    6.  If the signatures do not match, the request block is flagged as a high-threat payment forgery attempt and blocked immediately. Only upon cryptographic confirmation is the order updated to `paid` status.

---

## 2. Prevention of URL Editing / Privilege Escalation (IDOR)
*   **The Hazard**: A user altering transaction or route variables (e.g., changing `/orders/42` to `/orders/99`) in order to view, download, or verify payments belonging to another customer.
*   **Our Solution**: Combined JWT Identity Verification and Multi-Tenant Query Scope:
    1.  All payment verification and order history retrieval endpoints pass through active passport-style JWT session validation middleware (`requireAuth`). The session's associated customer identifier (`req.user.id`) is decoded from the cryptographic token cookie.
    2.  The SQL database queries look like:
        ```sql
        SELECT * FROM orders WHERE id = $1 AND user_id = $2
        ```
    3.  Because the database query directly filters on the *active decoded user session ID* (which the client cannot modify or override), horizontal traversal across order records is blocked.

---

## 3. Prevention of Wrong-User Payment Validation
*   **The Hazard**: Authenticated User A tries to pay for an order created by Authenticated User B by supplying User B's `orderId` to the payment verification route.
*   **Our Solution**: Pre-Transaction Record Guarding:
    1.  Upon receiving `orderId` at `/api/payments/verify`, the backend fetches the corresponding order record from PostgreSQL.
    2.  Before executing any hash calculation or external API contacts, the server checks if the order's registered `userId` matches the validated `req.user.id` from the secure HttpOnly JWT.
    3.  If they do not align perfectly, the server logs the incident and rejects the operation with a `403 Forbidden` status.

---

## 4. Prevention of SQL Injection in Payment Systems
*   **Our Solution**: Zero Unparameterized SQL Statements:
    1.  We utilize a modern, secure Object-Relational Mapper (Drizzle ORM) for all PostgreSQL interactions. Drizzle utilizes prepared queries natively.
    2.  All custom queries bind properties dynamically using structured input bindings (`eq()`, `and()`, `sql` query templates with parameterized parameters). Raw, unchecked string concatenation is strictly forbidden.

---

## 5. Prevention of Duplicate Payment Marking (Double-Spending / Idempotency)
*   **The Hazard**: A user clicks the pay button multiple times or submits payment verification requests repeatedly, attempting to double-receive rewards, loyalty credits, or cause database race conditions.
*   **Our Solution**: Safe Database Transaction Isolation and ID Checks:
    1.  The `orders` table tracks critical payment indicators, such as `paymentStatus` (e.g., `paid`, `unpaid`, `failed`), and the custom transaction identifier `razorpayPaymentId`.
    2.  When a verification request arrives, we execute the check under a transaction lock: we retrieve the current `paymentStatus`.
    3.  If `paymentStatus` is already marked as `paid`, the server terminates the process immediately with a clean `200 Success` (or skips updating), preventing duplicate loyalty point accumulations or state transition re-triggering.

---

## 6. Request Spam and Webhook Abuse Protection
*   **Our Solution**: Multi-Layer Rate-Limiting and Secret Handshakes:
    1.  **Rate Limiting**: Sliding window rate limits restrict payment attempts (`POST /api/orders` and `POST /api/payments/verify` are limited to a maximum of 4 request submissions per minute per IP address).
    2.  **Webhook Authentication**: All Razorpay webhooks (`/api/payments/razorpay/webhook/route.ts`) and Stripe webhooks must include verification signatures. Razorpay sends a `X-Razorpay-Signature` which is verified using `RAZORPAY_WEBHOOK_SECRET` before processing. Stripe webhooks utilize `stripe.webhooks.constructEvent()` with `STRIPE_WEBHOOK_SECRET`. Raw, unverified Webhook events are completely ignored.
